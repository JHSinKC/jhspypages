Try / Except / Else / Finally block
Try: this is the code you want to executre; logs INFO
Except: code jumps here and logs first error.
Else: executers if 'try' block succeeds
Finally: this code runs no matter what; close things down.

What is an Exception?
An exception is a signal that an error or other unusual condition has occurred.

Types of Errors?
An exception is a 'runtime error' happens during execution of a program.
A 'logical error' is when the data does not match the logical context.
A 'syntax or parsing error' occurs during coding or compile time.

What is an Error Handler?
A block of code that answers an error. If the 'try' block runs, the 'except' block is ingnored; if the 'try' block fails, the 'except' block(s) run.
Statements under the 'else' block run only when no exception is raised.
Statements in the optional 'finally' block will run every time no matter exception occurs or not.
['try' block]
try: number = int(raw_input("Enter a number between 1 - 10"))
['except' block]
except ValueError:
print "Inut error. Numbers only."
sys.exit()
['else' block]
else: doing_different_exception_handling
['finally' block]
finally: print('Goodbye, world!')

What is a 'logger'?
Loggers are the objects a developer usually interacts with. They are the main APIs that indicate what we want to log. Logger names are hierarchical. That is why I choose to use the module name as the logger name. The hierarchy starts with the root logger and descends from there.
Loggers, logging levels, handlers are configured at the application-level; do not configure loggers inside your code.

What is a 'log level'?
There are five levels of importance associated with logging events. Creating more levels is unhelpful, since assigning events to levels becomes subjective.
  • 50 - 'logger.critical()': Critical the application meltdown is imminent.
  • 40 - 'logger.error()': Error the application has stopped, due to the problem being logged.
  • 30 - 'logger.warning()': Warning the application is still executing, but may have problems in the near future.
  • 20 - 'logger.info()': Affirmation the application is running normally (e.g. "service has started" or "indexing run complete")
  • 10 - 'logger.debug()': Detailed information used in diagnosing problems.
  • 00 - NOTSET:

    • What is a 'log record'?
    Events are described by Log records; packages that the logging module uses to pass required information around. Log records contain information about the log level, the function where the log was requested, the string that was passed, arguments, call stack information, etc. LogRecord instances are created automatically by the Logger every time something is logged, and can be created manually via makeLogRecord().

    What is a 'log formatter'?
    16.6.7 LogRecord attributes. A Formatter is exactly what it sounds like. It formats a LogRecord. Formatter objects convert a LogRecord to (usually) a string which can be interpreted by either a human or an external system. Formatter objects configure the final order, structure, and contents of the log message. Formatters are in charge of serializing the metadata-rich LogRecord into a string.
    Using loggger.info("string template {}".format(argument)) should be avoided whenever possible in favor of logger.info("string template %s", argument). This is a better practice, as the actual string interpolation will be used only if the log will be emitted. Not doing so can lead to wasted cycles when we are logging on a level over INFO, as the interpolation will still occur.

    What is a 'log filter'?
    Filters allow for finer-grained control of which logs should be emitted. Multiple filters can be attached to both loggers and handlers. For a log to be emitted, all filters should allow the record to pass. Users can declare their own filters as objects using a filter method that takes a record as input and returns True/False as output. Filters can be used by Handlers and Loggers for more sophisticated filtering than is provided by levels. A Filter filters a LogRecord (I have never needed one). A filter object has a filter() method which is passed an event and which returns a value indicating whether the event is to be processed. Multiple filters can be associated with loggers and handlers, which both inherit from a base class Filterer having addFilter(), removeFilter() and filter() methods.
    The Filterer.filter() method calls the filter() method on all attached filters until all have seen the event, or until one of the filters rejects the event. If the event comes through unscathed, it is processed.

    What is a 'log handler'?
    16.8. 'logging.handlers' are responsible for dispatching the appropriate log messages (based on the log messages’ severity) to the handler’s specified destination (output). There are 13 standard 'handlers', sending log records into a defined output (e.g. files, syslog, HTTP servers, sockets).
    An application may send log messages to a log file, all log messages of error or higher to stdout, and all messages of critical to an email address. Three individual handlers are required; each handler is responsible for sending messages of a specific severity to a specific location.
    The standard logging module already comes with multiple built-in output handlers like:
    1. Output stream: StreamHandler, 'systemd' (stdout or stderr)
    2. File handler: FileHandler inherits from StreamHandler
    3. Files (watched):
    4. Rotating set of files (size): RotatingFileHandler switches from one file to the next when the current file reaches a certain size.
    5. Rotating set of files (time): TimedRotatingFileHandler switches from one file to the next at specified times.
    6. Email: SMTPHandler sends the events to designated e-mail addresses. SMTPHandlers are generally used to notify people about problems which need urgent handling.
    7. Streaming TCP socket: SocketHandler sends the events, via a socket, to a remote server listening on a TCP port.
    8. Streaming UDP socket: (similar to SocketHandler) DatagramHandler sends the events using UDP sockets. Lower overhead, lowerreliability.
    9. Web server: HTTPHandler
    10. SysLog (UNIX/Linux): SyslogHandler, MemoryHandler
    11. SysLog (Windows): NTEventLogHandler sends the events to an NT event log.
    12. Memory Buffer: When a trigger is received, MemoryHandler sends events to another handler to deal with.
    13. NULL Device: NullHandler is used in library code which uses logging to avoid misconfiguration messages when used in an application which doesn't configure logging.

    Most logging systems tell you what happened in your application, whereas eliot also tells you why it happened.
    eliot is a Python logging system that outputs causal chains of actions: actions can spawn other actions, and eventually they either succeed or fail. The resulting logs tell you the story of what your software did: what happened, and what caused it.
    Eliot works well within a single process, but can also be used across multiple processes to trace causality across a distributed system. Eliot is only used to generate your logs; you will still need tools like Logstash and ElasticSearch to aggregate and store logs if you are using multiple processes.
